# TECH BLOG WRITING SKILL
## Experience-based technology content that ranks, converts, and gets cited by AI engines

---

## Core Philosophy

> **Write like a practitioner, not a journalist.**
> Every tech blog post must answer: "Why should I trust THIS person on THIS topic?"

Three pillars every post must have:
1. **Experience** — first-hand usage, real results, actual problems encountered
2. **Technical depth** — specific version numbers, commands, configs, benchmarks
3. **Practical value** — reader must be able to DO something after reading

---

## CONTENT TYPES — Pick One Before Writing

### Type 1 — Experience Report
*"I used X for 30 days — here's what happened"*
- Personal narrative + data
- Honest pros AND cons
- Real screenshots or metrics if available
- Best for: tools, frameworks, platforms, services

### Type 2 — Technical Tutorial
*"How to do X step by step"*
- Numbered steps with code blocks
- Prerequisites clearly listed upfront
- Expected output shown at each step
- Common errors + fixes included
- Best for: setup guides, configurations, integrations

### Type 3 — Problem → Solution
*"I had X problem. Here's exactly how I fixed it"*
- Start with the exact error or symptom
- Walk through diagnosis
- Show the fix with code/config
- Explain WHY it works, not just what to do
- Best for: debugging guides, workarounds, performance fixes

### Type 4 — Tool Comparison
*"X vs Y — which should you use in 2026?"*
- Side-by-side feature table
- Use-case based recommendation
- Personal verdict at the end
- Best for: commercial intent keywords, affiliate content

### Type 5 — Roundup / Best Of
*"Best X tools for developers in 2026"*
- 5–10 items minimum
- Each with: what it does, who it's for, pricing, verdict
- Clear winner recommendation
- Best for: affiliate links, high commercial intent traffic

## PHASE 1 — PRE-WRITING RESEARCH (mandatory)

Never write before completing all steps.

### Step 1 — Search Intelligence (3 passes)
```
Pass 1: "[topic] 2026"                    → see who ranks and what format
Pass 2: "how to [action] [topic]"         → informational intent landscape
Pass 3: "[topic] review / experience"     → find content gaps
```

Extract from top 3 results:
- Word count (your target = their average × 1.2)
- Subheadings used (your H2s must cover all of these + more)
- What they got WRONG or skipped (your differentiation)
- People Also Ask questions (these become your H3s and FAQs)

### Step 2 — Experience Inventory
Before writing, list:
```
- Have I used this tool/tech personally? (yes/no)
- What specific problem did I solve with it?
- What version / date did I use it?
- What results did I get? (time saved, errors fixed, performance gain)
- What frustrated me about it?
- What surprised me?
```
If answer to first question is no — frame as "tested" or "researched" not "used daily."
Never fabricate experience. Write what you actually know.

### Step 3 — Keyword Selection
Primary keyword rules:
- One keyword per post — never two
- Must be in title within first 40 chars
- KD target: <40 for new sites
- Volume target: 100–5,000/month
- Long-tail always beats head terms at low DA

Tech keyword formula:
```
[technology name] + [specific action/problem] + [year]
Examples:
✅ "Next.js App Router caching guide 2026"
✅ "JWT token expiry error fix Node.js"
✅ "best IPS monitor for developers under $300"
❌ "JavaScript guide" (too broad)
❌ "monitors" (no intent)
```

## PHASE 2 — ARTICLE STRUCTURE

### Title Formula
```
Max 55 characters — hard limit
Format: [Primary keyword] [year] — [power word] [benefit]

Power words for tech: Complete, Exact, Fix, Fast, Free, Proven, Step-by-Step, Real

Examples:
✅ "JWT Decoder Guide 2026 — Fix Token Errors Fast"        (50 chars)
✅ "Next.js Caching Explained — Complete 2026 Guide"       (51 chars)
✅ "Best Dev Monitor 2026 — IPS vs VA Compared"            (46 chars)
❌ "A Complete and Comprehensive Guide to Understanding JSON Web Tokens in Next.js Applications"
```

### Meta Description Formula
```
140–155 characters exactly
Format: [keyword action] + [specific benefit] + [credibility signal]

Example:
"Fix JWT token expiry errors in Node.js in 3 steps.
Covers RS256, HS256, and clock skew issues. Tested on Node 20 + Express 5."
(152 chars)
```

### URL Slug Rules
```
- Lowercase only
- Hyphens between words, no underscores
- No stop words (a, the, and, in, for, of)
- Include primary keyword
- Max 5–6 words

✅ /blog/jwt-decoder-nodejs-guide
✅ /blog/best-monitor-developers-2026
❌ /blog/a-complete-guide-to-understanding-jwt-tokens-in-node-js
```

## PHASE 4 — WRITING STYLE RULES

### Voice
- Write in first person for experience posts ("I tested...", "When I built...")
- Write in second person for tutorials ("You'll need...", "Run this command...")
- Never write in third person ("developers should...") — too distant, too generic

### Technical Accuracy Rules
- Always include version numbers: "Node.js 20.11", not just "Node.js"
- Always include the year for best practices: "as of 2026"
- Always show expected output after code blocks
- Never say "simply" or "just" — these words frustrate readers who are stuck
- Never say "easy" — what's easy for you may not be for them

### Code Block Rules
```
- Every code block must have a language tag: ```javascript ```bash ```json
- Add a comment above complex lines explaining what it does
- Show the COMPLETE snippet, not partial — readers copy-paste
- If config file: show which file it belongs to as a comment on line 1
- Mark changed/important lines with a comment: // ← add this line
```

### Paragraph Rules
- Max 3–4 sentences per paragraph
- One idea per paragraph
- No walls of text — break with subheadings every 200–300 words
- Use bold for key terms on first use only — not repeatedly

### Tone Calibration by Post Type
| Post Type | Tone | Example opener |
|-----------|------|----------------|
| Tutorial | Calm, precise, instructional | "This guide walks you through..." |
| Experience report | Honest, conversational | "Three months ago I switched to..." |
| Problem/fix | Urgent, direct | "If you're seeing this error..." |
| Comparison | Balanced, analytical | "Both tools solve the same problem differently." |
| Roundup | Opinionated, confident | "After testing 12 monitors, only 3 are worth it." |

## PHASE 5 — GEO OPTIMISATION (get cited by AI engines)

For every article, implement all 5 rules:

### Rule 1 — Definition First
Within the first 150 words, include a crisp 2–3 sentence definition.
Format: "[X] is [what it is]. It works by [mechanism]. In 2026, [current state/relevance]."
AI engines extract this verbatim when answering "what is X."

### Rule 2 — Answer Boxes After Every H2
Immediately after each H2 heading, write a 40–60 word direct answer.
No preamble. No "in this section we will explore."
Just: the answer. AI engines use these as citations.

### Rule 3 — Named Technical Entities
Mention specific: version numbers, RFC standards, spec names, official documentation links.
"According to RFC 7519, JWT tokens must..." beats "JWT tokens should..." every time.
LLMs prefer pages that cite specific authoritative references.

### Rule 4 — Comparison Tables
At least one table per post. AI engines love extracting structured comparisons.
Always include a "Winner" or "Best for" column.

### Rule 5 — FAQ Schema Ready
Write FAQs as standalone question + answer pairs.
Each answer must make sense WITHOUT reading the rest of the article.
These become FAQPage JSON-LD schema entries.

Every published post must have ALL of these:

- [ ] "Last updated: [date]" near the title
- [ ] At least one first-person experience statement in first 200 words
- [ ] Minimum 2 outbound links to authoritative sources (MDN, RFC docs, official repos, gov sites)
- [ ] Specific version numbers mentioned at least twice
- [ ] At least one thing the author found frustrating or that didn't work perfectly (honesty = trust)
- [ ] No unverifiable claims without a source

Every post must:

**Link OUT to (minimum 3):**
- 2 related blog posts on wtkpro.site
- 1 relevant tool page on wtkpro.site (with descriptive anchor text)

**Be linked FROM (minimum 2 — update existing posts):**
- Find 2 existing posts where this topic is mentioned

## PHASE 8 — SCHEMA FOR TECH POSTS

Every tech blog post gets these 3 schema blocks:

### Article Schema (required)
```json
{
  "@context": "https://schema.org",
  "@type": "TechArticle",
  "headline": "[post title — max 110 chars]",
  "datePublished": "[ISO date]",
  "dateModified": "[ISO date]",
  "author": {
    "@type": "Person",
    "name": "Abu Sufyan",
    "url": "https://wtkpro.site/about"
  },
  "publisher": {
    "@type": "Organization",
    "name": "WebToolkit Pro",
    "url": "https://wtkpro.site"
  },
  "about": {
    "@type": "Thing",
    "name": "[main topic]",
    "sameAs": "[wikidata URL for topic]"
  }
}
```

### FAQPage Schema (required if FAQs present)
```json
{
  "@context": "https://schema.org",
  "@type": "FAQPage",
  "mainEntity": [
    {
      "@type": "Question",
      "name": "[question text]",
      "acceptedAnswer": {
        "@type": "Answer",
        "text": "[answer text — plain, no HTML]"
      }
    }
  ]
}
```

### HowTo Schema (for tutorial posts only)
```json
{
  "@context": "https://schema.org",
  "@type": "HowTo",
  "name": "[tutorial title]",
  "step": [
    {
      "@type": "HowToStep",
      "name": "[Step 1 name]",
      "text": "[Step 1 description]"
    }
  ]
}
```

## AFFILIATE CONTENT RULES (for product posts)

When writing posts with Amazon affiliate links:

- [ ] FTC disclosure at top: "This post contains affiliate links. I may earn a commission at no extra cost to you."
- [ ] Honest pros AND cons — never only positives
- [ ] Personal verdict must be genuine — if you haven't used it, say "based on specs and reviews"
- [ ] Price mentioned as "check current price →" not a fixed number (prices change)
- [ ] At least one alternative recommended (even if not affiliated)
- [ ] Affiliate link placed naturally — never forced
